# Kolte Patil Production Architecture: Access-Token Verification

When the Mobile App consumes the new `POST /projects/list` payload, it will transmit an `Access-Token` encoded in the headers. Based on the Post-Sale API Document, this token starts with `"eyJhbGci..."`, which guarantees that the company is utilizing **Stateless JSON Web Tokens (JWT)**.

Here is exactly how the backend verifies that string instantaneously:

### The Secret Key Handshake (Stateless Verification)
Because the token is a JWT, the backend **does not** physically need to ping the database or the legacy PHP servers to check if the user is real. The token itself is actually a cryptographically compressed block of JSON data containing the user's ID, digitally stamped with a mathematical "Signature".

**The Verification Flow:**
1. **The Request:** The Mobile App fires the request natively: `Access-Token: eyJh...`
2. **The Interception:** Our Node.js Express server intercepts the header immediately.
3. **The Math Check:** The Node.js server takes the token and attempts to apply a highly secure `JWT_SECRET` password (stored locally in our `.env` server variables) against the token's cryptographic signature.
4. **The Verdict:** 
   - If the password successfully decrypts the signature, the Node.js server mathematically *knows with 100% absolute certainty* that the token was generated by Kolte Patil's true authentication server, and it instantly allows the project payload to be pushed back to the mobile app.
   - If the token is corrupted, tampered with, or explicitly expired off the 3600-second timeline, the math fails violently, and our API safely returns the `401 Unauthorized: Invalid or expired access token` error documented by your manager.

### Implementation Requirement
To natively stitch our localized `/projects/list` microservice flawlessly into your enterprise ecosystem, all your backend engineering team has to do is copy the **exact same `JWT_SECRET` string** that the main Legacy OAuth API is using, and paste it identically into our Node.js `.env` configuration file!

---

### How One Secret Password Decrypts Thousands of Unique User Tokens
It can seem confusing how the exact same `JWT_SECRET` string in the `.env` file can somehow decrypt tokens belonging to millions of completely different mobile users. 

The `JWT_SECRET` is NOT the user's password! It is the **Server's Master Key** (an massive unguessable string like `aB9f@X51#LpQ!v3#1Z`).

When **User A** logs into the mobile app, the authentication server takes User A's data (like their email or User ID), and cryptographically scrambles it using that Master Key, turning it into the `eyJhb...` Access-Token string.
When **User B** logs in, the server uses that *exact same Master Key* to scramble User B's data into a totally different `eyJhb...` string.

Because your new `/projects/list` Node.js server also holds the exact same Master Key inside its `.env` file, it knows the mathematical algorithm required to unscramble every single one of those tokens. It can instantly crack them open and reveal whether it belongs to User A or User B, without ever needing to touch a database!

**Example Node.js Validation Middleware:**
```javascript
const jwt = require('jsonwebtoken');

function verifyMobileUser(req, res, next) {
    const token = req.headers['access-token'];
    
    try {
        // The magic happens here! One Server Master Password unlocks every user token.
        const decodedUserData = jwt.verify(token, process.env.JWT_SECRET);
        
        req.user = decodedUserData; // Instantly tells us exactly who the customer is!
        next();
    } catch (err) {
        res.status(401).json({ status_message: "Unauthorized: Invalid or expired access token" });
    }
}
```
